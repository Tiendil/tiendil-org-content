[Подводя итоги прошлого года]{post:results-of-2025-for-me-and-the-blog} я говорил, что агенты станут главной темой 2026. Последние месяца три я постепенно разбираюсь с использованием их для разработки ПО — можно обсудить промежуточный результат.

Нормального русскоязычного термина для coding agents пока нет, поэтому я буду использовать англоязычный.

Как я использовал coding agents:

- Рефакторил и немного писал код на Rust в экспериментальном движке для игровой логики.
- Использовал их для «стандартных» задач разработки при работе над [Feeds Fun](https://github.com/Tiendil/feeds.fun) (Python).
- Как трушный хакер решил, что чтобы разобраться с чем-то, надо это что-то написать с нуля, поэтому за месяц (принципиально) навайбкодил [Donna](https://github.com/Tiendil/donna) — утилиту/окрестратор для агентов (Python).

Поскольку агенты — штука новая, то:

- никто не знает как их использовать;
- ни у кого нет разностороннего опыта их использования — либо ты выбираешь один-два подхода и работаешь свою работу ими, либо ты бесконечно экспериментируешь, не нанося пользу.

Поэтому этот пост будет в форме субъективных заметок-тезисов, без попытки объективного анализа.

## Что такое (coding) agents

Для начала, чем агенты не являются:

- Это не «просто» [LLM](https://ru.wikipedia.org/wiki/Большая_языковая_модель), которая всё делает сама каким-то магическим образом.
- Это не [сильный/слабый ИИ](https://ru.wikipedia.org/wiki/Сильный_искусственный_интеллект), который действует как человек.

А сейчас будет ересь, приготовьтесь.

Агент — это «классическое не-ИИ ПО» — программная обвеска-оркестратор над внутренними и внешними инструментами, которая организует их совместную работу для достижения какой-то цели. Инструменты могут быть разными: внутренний код, консольные утилиты, сервисы с HTTP API, LLM, etc. Есть даже такая классика как [агентно-ориентированное программирование](https://ru.wikipedia.org/wiki/Агентно-ориентированный_подход).

Это хорошо заметно даже по фокусу разработки агентов, changelog-ам — их разработчики решают вопросы безопасного запуска инструментов, планирования, коммуникации между ними, взаимодействия с пользователем, etc. — вопросам связанным непосредственно с ИИ там уделяется довольно мало времени.

Концептуально, coding agent может даже не содержать ИИ компонентов:

- [Linter](https://en.wikipedia.org/wiki/Lint_(software)) с автоматическими исправлениями, например [isort](https://pycqa.github.io/isort/), вполне может считаться узкоспециализированным coding agent.
- [Github Actions](https://docs.github.com/en/actions) — это вот прямо агенты-агенты, они ещё и реактивные — реагируют на события в репозитории и автоматически что-то делают.
- С помощью [Make](https://ru.wikipedia.org/wiki/Make) стопудово можно создать простого агента, где обращение к LLM будет одним из этапов.

Никто не называл эти штуки агентами — «можно, а зачем?» (с).

Появление LLM вдохнуло новую жизнь в концепцию.

Во-первых, LLM позволили связывать инструменты на более высоком уровне абстракции. Если раньше программист должен был чётко прописывать [трансформацию схем данных]{post:infinity-of-data-schemas} между каждой парой инструментов для конкретной задачи, то теперь это делает LLM для любых задач и любых инструментов.

Во-вторых, LLM могут «понимать» (на приемлемом уровне) код, что позволяет не только связывать инструменты ими, но и делать минимальное планирование.

Эти две вещи сделали agent-like архитектуру более употребимой и актуальной, что потребовало вернуть термин в активный словарь.

## Мой контекст использования coding agents

Я использую агентов не ради использования агентов, а чтобы эффективнее делать свою работу. Поэтому у меня не было цели попробовать все возможные подходы, или весь возможный софт.

Когда я встречаяю что-то новое, обычно происходит следующая цепочтка событий: «о, интересно» —> читаю документацию —> не понимаю как оно может улучшить мою жизнь —> иду дальше.

Поэтому у меня сейчас довольно аскетичный сетап плюс небольшая табличка в notion со списком инструментов/софта, которые я бы хотел попробовать при случае.

Мой сетап:

- [Codex CLI](https://developers.openai.com/codex/cli) как coding agent.
- [Donna](https://github.com/Tiendil/donna) — инструмент для высокоуровневого планирования.
- [ast-grep](https://github.com/ast-grep/ast-grep) — инструмент для более оптимальной навигации по коду и его трансформации.
- Специфичные для проекта [большие спецификации](https://github.com/Tiendil/donna/blob/c1e3f3e1354f60263c220fef5e7b2754deb09190/.donna/project/core/error_handling.md) (частично в формате Donna) вместо skills.

Это не значит что я не смотрю на новые вещи и не пробую их, это значит что они не прописываются в моём рабочем процессе.
